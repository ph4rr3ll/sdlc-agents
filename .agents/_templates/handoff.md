# HANDOFF REPORT TEMPLATE

Каждый специализированный агент завершает работу отчётом в этом формате. Менеджер использует отчёт для синтеза, обновления `STATE.json` и выбора следующего шага.

В **Native mode** отчёт возвращается как последнее сообщение субагента.
В **File mode** отчёт сохраняется в `.agent_space/[TASK_ID]/handoffs/[NN]_[agent_id]_report.md`.

---

## ОБЯЗАТЕЛЬНАЯ СТРУКТУРА ОТЧЁТА

```markdown
# HANDOFF: [agent_id] → manager  (step NN)

## SUMMARY
[3–5 предложений: что сделал, ключевой результат, итог. Без процесса размышлений.]

## ARTIFACTS
Список созданных или изменённых файлов с краткими аннотациями:
- `[WORKSPACE_RELATIVE_PATH]` — [что это]
- `[WORKSPACE_RELATIVE_PATH]` — [что это]

## EVIDENCE
**Conditional секция (Слой 2 anti-hallucination).**

- **Обязательна** для агентов с верифицируемыми claims: `coder_backend`, `coder_frontend`, `reviewer`, `qa`. Минимум одна `[VERIFIED:...]` запись — иначе `validate.py` отклонит handoff.
- **Опциональна** для `shaper`, `devops`, `manager`. Заполняй, только если есть verifiable claims (запуск инструмента, цитирование существующего файла). Иначе секцию можно удалить.

Формат — список с явными маркерами:
- `[VERIFIED:cmd]` <команда, которую ты прогнал> → <краткий вывод или путь к логу>
- `[VERIFIED:file]` <путь>:<строки> → <что прочитано/проверено>
- `[ASSUMED]` <утверждение> — <обоснование, почему допущено без проверки>

`[ASSUMED]` допустимы, но менеджер их подсветит (даже если EVIDENCE опциональна) и может потребовать дополнительной проверки.

Примеры:
- `[VERIFIED:cmd]` `uv run pytest apps/core/tests/test_health.py -v` → 5 passed (лог: `artifacts/test_runs/01_health.txt`)
- `[VERIFIED:file]` `REQUIREMENTS.md:15-23` → AC-01-1, AC-01-2 процитированы дословно
- `[VERIFIED:cmd]` `grep -n "class UserService" apps/core/services/` → найден `user.py:42`, компонент C-01 опирается на эту существующую сущность
- `[ASSUMED]` версия деплоя берётся из git-тега — не проверял CI-конфиг, опираюсь на упоминание в брифе

## DECISIONS
Ключевые решения, влияющие на дальнейшие этапы. Формат:
- **Решение**: [что выбрано]
  **Обоснование**: [почему]
  **Альтернативы**: [что отвергнуто и почему]

## OPEN_QUESTIONS
[Опционально. Вопросы, на которые ты не смог ответить и которые влияют на следующий этап. Если вопросов нет — удали секцию.]

## BLOCKERS
[Опционально. Препятствия, требующие пользователя или другого агента. Если блокеров нет — удали секцию.]

## RECOMMENDED_NEXT
- **Агент**: [agent_id или `manager`, если нужен синтез]
- **Что передать**: [краткий перечень инпутов]
- **Стадия**: [предлагаемое значение STATE.json.current_stage]

## STATE_PATCH
Фрагмент JSON для применения к STATE.json. Менеджер валидирует и сливает.
```json
{
  "current_stage": "[NEW_STAGE]",
  "active_agent": "[NEXT_AGENT_OR_MANAGER]",
  "updated_at": "[ISO8601]",
  "artifacts": {
    "[logical_name]": "[path]"
  },
  "history_append": {
    "step": NN,
    "timestamp": "[ISO8601]",
    "from_agent": "[YOUR_AGENT_ID]",
    "to_agent": "manager",
    "stage": "[STAGE]",
    "summary": "[короткая сводка]",
    "artifacts": ["[path1]", "[path2]"]
  }
}
```
```

## ПРИНЦИПЫ ОТЧЁТНОСТИ

1. **Не повторяй содержимое артефактов.** Результат — файл, ссылайся на путь, не копируй контент в отчёт.
2. **Фиксируй решения, а не процесс.** "Выбрал PostgreSQL потому что нужна транзакционность" — да. "Сначала думал про MySQL, потом..." — нет.
3. **Чётко разделяй "сделано" и "осталось".** Open questions и blockers — отдельные секции, не размытые в summary.
4. **Не предписывай менеджеру.** RECOMMENDED_NEXT — рекомендация, не указание. Финальное решение за менеджером.
5. **STATE_PATCH — минимальный.** Только реально изменившиеся поля + одна запись в history_append.
