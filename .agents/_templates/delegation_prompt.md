# DELEGATION PROMPT TEMPLATE

Менеджер использует этот шаблон при делегировании задачи специалисту. Заполни все обязательные секции, опциональные удаляй, если они не релевантны.

В **Native mode** заполненный бриф передаётся как промпт субагенту.
В **File mode** бриф сохраняется в `.agent_space/[TASK_ID]/handoffs/[NN]_[agent_id]_brief.md`, где `NN` — двузначный номер шага из `STATE.json.history`.

---

## ОБЯЗАТЕЛЬНАЯ СТРУКТУРА БРИФА

```markdown
# BRIEF: manager → [agent_id]  (step NN)

## ROLE_ACTIVATION
Ты вызван как `[agent_id]`. Действуй согласно своему манифесту в `.agents/[agent_id].md`. Если ты запущен с этим брифом — НЕ читай `STATE.json` для определения задачи; вся нужная информация уже здесь. Используй `STATE.json` только для контекста (история, артефакты других этапов), если это явно требуется.

## TASK_GOAL
[1–3 предложения. Цель ИМЕННО ЭТОГО делегирования, а не всей задачи проекта. Что должен сделать этот агент.]

## WORKSPACE_CONTEXT
- `TASK_ID`: [идентификатор]
- `WORKSPACE`: `.agent_space/[TASK_ID]/`
- Исходная задача пользователя: [1–2 предложения]
- Текущая стадия: [значение из STATE.json.current_stage]

## INPUTS
Артефакты, которые ты ДОЛЖЕН прочитать перед началом:
- `[ПУТЬ_1]` — [что это, зачем читать]
- `[ПУТЬ_2]` — [что это, зачем читать]

## PRIOR_KNOWLEDGE
[Опционально. Что уже решено и не нуждается в перепроверке. Экономит контекст и предотвращает повторную работу. Если нечего сказать — удали секцию.]

## TECH_STACKS
[Опционально. Только для coder_*. Список stack-файлов, которые ты должен загрузить:]
- `.agents/stacks/[stack_id].md`

## EXPECTED_OUTPUT
- **Артефакт(ы)**: [какие файлы создать, по каким путям, в каком формате]
- **Handoff-отчёт**: завершающее сообщение в формате `.agents/_templates/handoff.md`

## CONSTRAINTS
- [Что НЕЛЬЗЯ делать в рамках этого шага]
- [Границы роли: чужие артефакты не трогать]
- [Бюджет: количество файлов / объём кода, если применимо]

## RETURN_FORMAT
Закончи работу handoff-отчётом со следующими полями (см. `.agents/_templates/handoff.md`):
- SUMMARY (3–5 предложений)
- ARTIFACTS (пути)
- **EVIDENCE** — обязательно для `coder_backend`, `coder_frontend`, `reviewer`, `qa` (минимум одна `[VERIFIED:...]` запись — иначе `validate.py` отклонит handoff). Опционально для прочих ролей; заполняй только если есть verifiable claims.
- DECISIONS (ключевые решения и обоснования)
- OPEN_QUESTIONS (если есть)
- BLOCKERS (если есть)
- RECOMMENDED_NEXT
- STATE_PATCH (фрагмент JSON для применения к STATE.json)
```

## ПРИНЦИПЫ ФОРМИРОВАНИЯ БРИФА

1. **Брифуй как нового коллегу.** Агент не видит истории диалога, не помнит, что ты решил с предыдущим специалистом. Всё нужное — в брифе или в файлах по указанным путям.
2. **Не передавай лишнего.** Каждый параграф должен влиять на решение агента. Сомневаешься — выкинь.
3. **Не делегируй понимание.** НЕ пиши "разберись с задачей". Сформулируй конкретный наблюдаемый результат.
4. **Фиксируй формат вывода.** Чем строже формат — тем легче сводить результаты разных агентов.
5. **Один бриф = одна цель.** Если задач две — два делегирования (последовательных или параллельных).
6. **Указывай явные пути.** Никаких "посмотри где-то в проекте" — конкретный путь.
7. **Не пересказывай содержимое файлов.** Дай путь, агент прочитает сам.
