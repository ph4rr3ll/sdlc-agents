# Локальное тестирование команды агентов

Памятка по обкатке мультиагентной команды на локальной модели через Kilo Code (VS Code). Под Apple Silicon + MLX.

## TL;DR

1. Поднять MLX-сервер с моделью.
2. Настроить Kilo Code на локальный endpoint.
3. Прогнать смоук-тесты от простого к сложному.
4. Не ждать чудес от 4-битной квантизации в первом прогоне — это нормально.

## Что понадобится

- macOS на Apple Silicon (M1/M2/M3/M4).
- VS Code + Kilo Code extension.
- LM Studio: https://lmstudio.ai
- Python 3.10+ (только для скриптов валидации, не для сервера).
- ~25 ГБ свободного места под модель.
- Этот репозиторий открыт в VS Code как workspace.

---

## Шаг 1. Поставить LM Studio

1. Скачай с https://lmstudio.ai и установи.
2. Открой LM Studio. Стартовый wizard можно пропустить.
3. Settings (⚙) → Runtime: убедись, что **MLX** runtime активен (для Apple Silicon включается по умолчанию).

## Шаг 2. Скачать модель

В LM Studio: вкладка `Discover` (🔍) → ищи модель по имени (например, `Qwen2.5-Coder-32B-Instruct`) → в списке квантизаций выбери **MLX 4bit** → Download.

LM Studio покажет совместимые варианты квантизации; для M4 Pro 48GB бери MLX (не GGUF — MLX на Apple Silicon заметно быстрее).

Какую модель брать конкретно — см. блок «Модели на M4 Pro 48GB» ниже.

## Шаг 3. Загрузить модель и поднять сервер

1. Вкладка `Chat` (💬) → в верхней панели выбери загруженную модель → подожди, пока она прогрузится в память (RAM-индикатор покажет загрузку).
2. (Опционально) В правой панели → `Context Length` → выставь 32768+ для длинных агентских контекстов (но не больше, чем влезает в память — LM Studio предупредит).
3. Вкладка `Developer` (</>): включи тумблер **Status: Running** — это поднимает локальный OpenAI-совместимый сервер.
4. Эндпоинт по умолчанию: `http://localhost:1234/v1`.

Проверка живости:
```bash
curl -s http://localhost:1234/v1/models | jq .
```
Должен вернуть список с твоей загруженной моделью.

## Шаг 4. Настроить Kilo Code

В VS Code:
1. Command Palette → `Kilo Code: Open Settings`.
2. Provider: **OpenAI Compatible**.
3. Base URL: `http://localhost:1234/v1`.
4. API Key: любая непустая строка (LM Studio не проверяет).
5. Model: имя, как показано в `LM Studio → Developer → API → "Model identifier"` (часто `lmstudio-community/Qwen2.5-Coder-32B-Instruct-MLX-4bit` или похожее).
6. (Опционально) `max_tokens` ответа = 4096 — снижает риск переполнения контекста.
7. Сохранить.

Затем: Command Palette → `Kilo Code: Reload Config`. Должны появиться все 10 агентов из `.kilo/agents/`.

## Модели на M4 Pro 48GB

С 48 ГБ unified memory под модель доступно ~36–40 ГБ (остальное забирают macOS и приложения). Грубо: 4-битная квантизация ≈ 0.5 байта на параметр, плюс KV-cache 2–4 ГБ на длинный контекст.

### Tier S — рекомендуемый дефолт
**`Qwen2.5-Coder-32B-Instruct-MLX-4bit`** — ~18 ГБ.
Coder-специализированный, лучшее качество кода в этом размере. Жёсткая дисциплина формата (это критично для агентов). На M4 Pro ожидаемо ~25–35 tok/s. Контекст до 32K без проблем; до 128K — если хватит RAM.

### Tier A — альтернативы
- **`Qwen3-30B-A3B-Instruct-MLX-4bit`** — ~16 ГБ.
  MoE с ~3B активных параметров. Быстрее (~60–80 tok/s), но менее заточен под код. Хорош для shaper'а и для итеративной отладки пайплайна.
- **`Qwen2.5-Coder-32B-Instruct-MLX-8bit`** — ~34 ГБ.
  Если 4bit плохо держит формат — 8bit даёт заметно лучшую дисциплину. На грани твоей памяти; контекст придётся сократить до ~16K.
- **`DeepSeek-Coder-V2-Lite-Instruct-MLX-4bit`** (16B MoE) — ~9 ГБ.
  Быстрый coder, чуть слабее Qwen2.5-Coder-32B по качеству, но отличный для быстрой отладки пайплайна.

### Tier B — если 32B слишком медленный
- **`Qwen2.5-Coder-14B-Instruct-MLX-4bit`** — ~8 ГБ. Быстро, качество чуть ниже.
- **`Qwen2.5-Coder-7B-Instruct-MLX-8bit`** — ~7 ГБ. Запасной вариант.

### Что НЕ брать
- **70B-модели** даже в 4bit — на грани памяти, KV-cache съест остаток, придётся резать контекст до 4–8K.
- **Heretic/abliterated** варианты — деградирует format discipline, см. отдельное рассуждение.
- **GGUF-варианты** — Llama.cpp на macOS работает, но MLX-runtime в LM Studio быстрее на Apple Silicon. Выбирай MLX.

### Практический совет
Начни с **`Qwen2.5-Coder-32B-Instruct-MLX-4bit`**. Прогон Smoke 0 (ниже) показывает, держит ли модель формат. Не держит — сначала попробуй `8bit` той же модели; всё ещё не держит — переходи на dense `Qwen2.5-32B-Instruct` (не Coder, но строже по формату).

## Smoke-тесты (по нарастающей)

### Smoke 0 — модель в принципе пригодна (one-shot quality check)

**До любых агентских прогонов** прогоните этот тест в чате LM Studio (или Kilo Code с любым agent'ом). Один запрос, один ответ — оценка занимает 5 минут и сразу показывает, тянет ли модель агентскую работу.

Скопируй промпт ниже целиком и пошли модели:

````
Ты — coder_backend. Тебе дан бриф ниже. Выполни его строго.

# BRIEF: manager → coder_backend  (step 07)

## TASK_GOAL
Реализовать функцию `sum_unique_integers(values: list[int]) -> int`, которая возвращает сумму уникальных целых чисел из списка (каждое уникальное значение учитывается один раз).
Пример: `sum_unique_integers([1, 2, 2, 3, 3, 3])` → `6`.
Пустой список → `0`.

## CONSTRAINTS
- Python 3.12+, явная типизация всех публичных сигнатур.
- Только стандартная библиотека.
- ЗАПРЕЩЕНО использовать `set()` для устранения дубликатов. Реализуй сам, проходя по списку.
- Минимум 4 теста на pytest: пустой список, все уникальные, все дубликаты, отрицательные числа.
- Никаких приветствий, объяснений сверху или снизу.

## RETURN_FORMAT
Ответ строго в этом порядке, без префиксов и постфиксов:

1. JSON-блок в ```json``` с полями:
   - `files_changed`: список из 2 имён файлов
   - `tests_added`: число
   - `tactical_decisions`: список строк по 1–2 предложения

2. Раздел `## function` — fenced Python-блок с реализацией функции.
3. Раздел `## tests` — fenced Python-блок с тестами на pytest.

ВСЁ. Никаких других секций. Никакого «вот мой ответ» в начале или «надеюсь, помог» в конце.
````

**Оценочный чек-лист (8 пунктов):**

| # | Критерий | Pass / Fail |
|---|---|---|
| 1 | Ответ **начинается** с JSON-блока (никакого «Конечно!» в первой строке) | ✅ / ❌ |
| 2 | JSON парсится (см. скрипт ниже) | ✅ / ❌ |
| 3 | Порядок секций: JSON → `## function` → `## tests`, без других | ✅ / ❌ |
| 4 | Функция реализована **без `set()`** | ✅ / ❌ |
| 5 | Логика корректна: `[1,2,2,3,3,3]` → 6 и `[]` → 0 (визуально) | ✅ / ❌ |
| 6 | ≥4 теста, покрывающих ВСЕ 4 заявленных кейса | ✅ / ❌ |
| 7 | Тесты — реальные `assert`, не `pass` или `assert True` | ✅ / ❌ |
| 8 | Никакого приветствия, эпилога или комментариев вне блоков | ✅ / ❌ |

Скрипт проверки JSON (сохрани ответ в `/tmp/r.md`, прогони):
```bash
python3 -c "
import re, json
t = open('/tmp/r.md').read()
m = re.search(r'\`\`\`json\s*(.*?)\`\`\`', t, re.DOTALL)
print('OK:', json.loads(m.group(1))) if m else print('NO JSON BLOCK')
"
```

**Вердикт:**
- **8/8** — модель пригодна, переходи к Smoke 1–5.
- **6–7/8** — пограничная. Попробуй ужесточить sampling: `temperature=0.1`, `top_p=0.9`. Если не помогло — возьми `8bit` вариант той же модели.
- **≤5/8** — не тянет агентский пайплайн. Смени модель (см. блок выше).

### Smoke 1 — модель отвечает
В чате Kilo Code (с любым agent'ом):
```
Ответь одним словом: 2+2=?
```
**Ожидание**: `4`. Если многословно или с emoji — сигнал, что модель плохо держит constrained-вывод.

### Smoke 2 — JSON-дисциплина
```
Выведи СТРОГО JSON без markdown-обёрток и комментариев:
{"status": "ok", "count": 3}
```
**Ожидание**: ровно эта строка. Если обернёт в ``` ``` или добавит «вот ваш JSON:» — заметь это, дальше будет влиять на работу с `STATE_PATCH`.

### Smoke 3 — менеджер инициализирует задачу
1. В agent picker (верхняя панель Kilo Code) выбери **manager**.
2. В чате:
   ```
   Инициализируй задачу TEST-001: вывести "Hello, world" в консоль через Python-скрипт.
   ```
3. **Ожидание**: созданы `.agent_space/TEST-001/STATE.json`, `handoffs/`, `artifacts/`. Менеджер говорит, что подготовил бриф для shaper.
4. **Валидация**:
   ```bash
   python3 -c "import json; s=json.load(open('.agent_space/TEST-001/STATE.json')); print(s['current_stage'], s['active_agent'])"
   # Ожидание: SHAPING shaper   (или INIT manager, если менеджер не успел перевести стадию)
   ```
5. Открой `handoffs/01_analyst_brief.md` — структура должна совпадать с шаблоном `.agents/_templates/delegation_prompt.md`.

### Smoke 4 — один специалист (shaper)
1. **Новая сессия чата** (важно). В Kilo Code: New Conversation.
2. В agent picker выбери **shaper**.
3. В чате:
   ```
   Возьми активную задачу.
   ```
4. **Ожидание**: аналитик найдёт `.agent_space/TEST-001/`, прочитает бриф, создаст `REQUIREMENTS.md`, напишет `02_analyst_report.md`.
5. **Валидация**:
   ```bash
   # Файлы существуют?
   ls .agent_space/TEST-001/artifacts/REQUIREMENTS.md .agent_space/TEST-001/handoffs/02_analyst_report.md

   # JSON STATE_PATCH парсится?
   python3 <<'PY'
   import re, json
   text = open('.agent_space/TEST-001/handoffs/02_analyst_report.md').read()
   m = re.search(r'```json\s*(.*?)\s*```', text, re.DOTALL)
   if not m:
       print('FAIL: нет json-блока')
   else:
       try:
           json.loads(m.group(1))
           print('OK: JSON валидный')
       except json.JSONDecodeError as e:
           print(f'FAIL: {e}')
   PY
   ```

### Smoke 5 — мини-цикл до планирования
Прогон: `manager → shaper → manager` (без `CODING/REVIEW/QA`).

**Каждый специалист — новой сессией чата.**

После shaper в `STATE.json` должно быть:
- `current_stage: CODING`
- 2 шага в `history` (manager→shaper, shaper→manager)
- Артефакты `requirements`, `arch_plan`, `tasks` все три присутствуют по своим путям.
- `tech_stacks` в STATE.json — массив выбранных стеков.

Если до этого момента модель не «сорвалась» по формату — пригодна. Дальше можно подключать `coder_backend` на реальной задаче в реальном проекте.

## Что считать «работает»

| Критерий | Проверка |
|---|---|
| JSON в `STATE_PATCH` парсится | `python3 -c "import json; ..."` |
| Файлы создаются по объявленным путям | `ls` |
| Агент не предлагает чужую работу (shaper не пишет код, manager не проектирует) | Чтение отчёта |
| Handoff содержит все обязательные секции | Сравнить с `_templates/handoff.md` |
| `current_stage` в `STATE.json` меняется корректно | `jq .current_stage .agent_space/.../STATE.json` |

## Красные флаги (особенно при heretic / 4-bit)

- Аналитик предлагает архитектуру или код.
- Менеджер сам пишет реализацию.
- Coder игнорирует stack-файл (`pip install` вместо `uv add`, default exports там, где запрещены).
- В JSON `STATE_PATCH` — комментарии `//` или trailing commas.
- Агент «срывается с роли» в середине ответа («Кстати, расскажу про…»).
- Игнорирование `CONSTRAINTS` манифеста («запрещено модифицировать рабочий код» — модифицирует).
- Выдуманные пути файлов (которых нет в манифесте/брифе).

Если 3+ флага из 7 на смоук-тестах 3–5 — модель не тянет, см. ниже.

## Если не работает

### Контекст переполняется
Симптомы: модель отвечает невпопад, повторяется, теряет инструкции.
- Сократи `max_tokens` ответа в Kilo Code до 2048.
- В LM Studio: правая панель чата → `Context Length` → увеличь (если позволяет память) или сократи (если упёрся в RAM).
- Дроби задачи мельче в `shaper.md` — атомарные T-XX на 1 файл.

### JSON ломается
- Добавь в бриф фразу: «JSON выводи только в block-fence ` ```json `, без trailing comma и комментариев, без текста после блока».
- Подключи grammar-constrained sampling — требует другого серверного движка (llama.cpp с `--grammar`, или outlines поверх mlx).

### Агент путает свою роль
- Проверь, что выбран правильный agent перед запросом (agent picker в верхней панели Kilo Code).
- Каждого специалиста запускай **новой сессией чата** (`Kilo Code: Reset Conversation` или просто New Conversation).
- Если игнорирует `CONSTRAINTS` — добавь явное в брифе: «**Запрещено**: <конкретное действие>. Если нарушаешь — это BLOCKER».

### Модель в принципе не тянет
См. блок **«Модели на M4 Pro 48GB»** выше — там основные варианты с обоснованием. Если ни один не подходит:

1. Возьми **8bit-вариант** той же модели — лучше формат, в 2× больше памяти.
2. Попробуй **dense** вместо MoE — `Qwen2.5-32B-Instruct` (не Coder, но строже формат).
3. **Облачная модель** через OpenRouter / DashScope / Anthropic — если локальные не справляются с дисциплиной, разумно взять managed API на сложные задачи и оставить локальную для лёгких.

### Heretic-симптомы конкретно
Если уже используешь heretic-версию и видишь срывы с роли / игнорирование constraints → переходи на обычный Instruct. Refusals на наших задачах не триггерятся (мы не пишем эксплойты), а format discipline у Instruct заметно лучше.

## Отладка

### Логи сервера
LM Studio → вкладка `Developer` → нижняя панель `Logs`. Видно каждый запрос (с полным телом) и ответ. Если запросы приходят, но ответы странные — проблема в модели, не в подключении.

### Логи Kilo Code
Output Panel → Kilo Code. Видны полные запросы и ответы. Полезно, когда модель ответила правильно, а Kilo Code обработал криво.

### Артефакты на диске — главный отладчик
Все ходы фиксируются в `.agent_space/<TASK_ID>/`. Что записано, в каком формате, в какой последовательности — всё видно. Сравнивай с эталонным `.agent_space/EXAMPLE-001/`.

### Откат к чистому состоянию
```bash
rm -rf .agent_space/TEST-*
```

## Резюмирующие правила для локального прогона

1. **Только обычный Instruct.** Heretic — не для агентских задач.
2. **Только File mode.** Native изоляции у тебя нет, не пытайся её эмулировать в одной сессии.
3. **Каждая роль — новая сессия чата.** «Помнит из прошлого» — не работает на 4-битной квантизации.
4. **Дроби задачи мельче.** Atomic > monolithic. Tech_lead должен это поддерживать.
5. **Прогон поэтапно.** Смоук 1 → 2 → 3 → 4 → 5. Не пытайся сразу гонять полный цикл с coder + reviewer + qa.
6. **`.agent_space/EXAMPLE-001/`** — твой эталон. Сравнивай форматы и содержание с ним при подозрениях.
