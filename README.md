# Multi-agent SDLC Team

> Готовая, портативная команда специализированных AI-агентов, проходящая полный цикл разработки — бизнес-анализ → проектирование → декомпозиция → разработка → ревью → тестирование → документация. Работает в Claude Code, Kilo Code v7.3+, Qwen Code и OpenAI Codex CLI.

---

## TL;DR

**Менеджер-оркестратор** принимает запрос пользователя, декомпозирует и поручает специалистам через шаблонные брифы. Каждый **специалист** работает в **изолированном контексте** со своим манифестом, обязан подтвердить выводы фактами в секции `EVIDENCE`, и возвращает структурированный handoff-отчёт. Перед каждым переходом стадии работают **три слоя защиты от каскадных галлюцинаций**: детерминированный валидатор, требование явного EVIDENCE, и опциональный human-gate на подтверждение от пользователя.

Команда — это **набор файлов**, не код. Кладёшь её в любой проект, добавляешь тонкий адаптер под выбранный инструмент — и менеджер начинает работать.

## Концепция

### Mental model

```
                    [user]
                       │
                       ▼
                  ┌─────────┐
                  │ MANAGER │  оркестратор: декомпозиция, брифы, синтез
                  └────┬────┘
                       │ Task / file brief
            ┌──────────┼──────────┐
            ▼          ▼          ▼
       ┌────────┐ ┌──────────┐ ┌──────────┐
       │ shaper │ │ coder_*  │ │ reviewer │  ...и так далее: qa, devops (conditional)
       └────────┘ └──────────┘ └──────────┘
       свой свежий контекст у каждого вызова
```

- **Менеджер** не пишет код, не проектирует, не формулирует требования. Он маршрутизирует.
- **Каждый специалист** получает свежий контекст в каждом вызове (Native mode) или явный бриф через файловый протокол (File mode).
- **Контракт коммуникации** — шаблон `delegation_prompt.md` для брифов, `handoff.md` для отчётов, `STATE.json` как журнал.

### Гибридная оркестрация

Команда работает в двух режимах в зависимости от возможностей инструмента:

- **Native mode** — инструмент имеет нативный механизм изолированных субагентов (`Task` в Claude Code, `Task` в Kilo Code v7.3+ с `task: allow`). Менеджер вызывает специалистов напрямую, каждый — в свежем контексте.
- **File mode** — нативной изоляции нет (Codex CLI, Qwen Code, старые версии). Передача управления — через `STATE.json` и `handoffs/`. Каждый специалист запускается **новой сессией** инструмента и читает свой бриф из файла.

Манифесты не различаются — один и тот же агент работает в обоих режимах.

### Три слоя защиты от каскадных галлюцинаций

При 6-ступенчатом пайплайне даже 10% риск галлюцинации на шаг компонуется до ~47% вероятности мусора на выходе. Три слоя по нарастающей строгости:

| Слой | Что делает | Кто исполняет | Файл |
|---|---|---|---|
| **1** | Детерминированно проверяет handoff: JSON-схема, существование путей, кросс-валидация ID, наличие `[VERIFIED]` в EVIDENCE, опционально перепрогон тестов. Скрипт не галлюцинирует. | Менеджер запускает в `SYNTHESIS_PROTOCOL` обязательно. | `.agents/_tools/validate.py` |
| **2** | Требование явных доказательств в handoff. Маркеры `[VERIFIED:cmd]` / `[VERIFIED:file]` / `[ASSUMED]`. Минимум одна `[VERIFIED]` — иначе валидатор отклоняет. | Специалист заполняет в каждом отчёте. | `.agents/_templates/handoff.md` (секция EVIDENCE) |
| **3** | Человек подтверждает каждый переход стадии. Toggleable per-transition. Ловит семантические ошибки (scope creep, неверный архитектурный выбор), которые скрипт не может проверить. | Пользователь. | `.agents/_config/gates.yaml` + `HUMAN-GATES.md` |

Слои 1+2 — гигиена, дёшевы. Слой 3 — стратегический контроль, можно отключать по переходам в `gates.yaml`.

## Команда агентов

| agent_id | Роль | Когда вызывается | Основной выход |
|---|---|---|---|
| `manager` | Оркестратор: декомпозиция, брифы, синтез | Всегда первым; между шагами для маршрутизации | поддерживает `STATE.json`, формирует брифы |
| `shaper` | Shaping: требования + архитектура + декомпозиция в один проход | После `INIT` | `REQUIREMENTS.md` + `ARCH_PLAN.md` + `TASKS.md` |
| `coder_backend` | Серверный код | На `CODING` для backend-задач | код + `IMPL_NOTES_T-XX.md` |
| `coder_frontend` | Клиентский код (база HTML/CSS/JS вшита; фреймворк — через стек) | На `CODING` для frontend-задач | код + `IMPL_NOTES_T-XX.md` |
| `reviewer` | Code review диффа | После каждой `CODING` задачи | `REVIEW_T-XX.md` (вердикт + findings) |
| `qa` | Трассировка AC → тесты, регрессия, NFR | После `REVIEW` clean | тесты + `QA_REPORT.md` |
| `devops` *(conditional)* | CI/CD, инфра, секреты, rollback-план | Только если `TASKS.md` от shaper содержит задачи с `Assignee: devops` | `DEVOPS_NOTES.md` |

### Доступные стеки

В `.agents/stacks/`:

- **`backend_python_django`** — Django 5.x + uv + DRF + pytest-django.
  Услуги/селекторы вместо fat models; signals — анти-паттерн; mocks только на границах внешних систем.
- **`frontend_react`** — React 19 + TypeScript strict + Vite + pnpm + TanStack Query.
  Feature-based layout; запрет axios и runtime CSS-in-JS; обязательный `eslint-plugin-jsx-a11y`.

Стек подключается через `tech_stacks` в брифе. Базовая экспертиза (HTTP/REST/SQL для backend; HTML/CSS/JS/a11y/perf для frontend) — внутри манифеста coder'а, не в стеке.

## Жизненный цикл задачи

```
INIT
  │  manager создаёт .agent_space/[TASK_ID]/{STATE.json, handoffs/, artifacts/}
  ▼
SHAPING    ─┐
  │ shaper │ → REQUIREMENTS.md + ARCH_PLAN.md + TASKS.md в один проход
  ▼        │
CODING     ├─ каждый переход:
  │ coder_*   1. validate.py
  ▼        │  2. EVIDENCE проверка
REVIEW     │  3. human-gate (если включён)
  │ reviewer  4. apply STATE_PATCH
  ▼        │  5. брифинг следующего
TESTING    │
  │ qa     │
  ▼       ─┘
DONE
```

`devops` подключается параллельно с `coder_*` в той же wave, если в `TASKS.md` от shaper есть задачи с `Assignee: devops`.

Цикл `REVIEW ⇄ CODING` возможен при `CHANGES_REQUESTED`; `TESTING → CODING` — при провале QA. `BLOCKED` — любой агент может вернуть, тогда менеджер останавливается и обращается к пользователю.

## Важные допущения и ограничения

### Что система НЕ решает

1. **Семантические ошибки в архитектуре.** Если `shaper` выбрал неправильный паттерн (например, синхронные вызовы там, где нужны очереди), `validate.py` это не поймает. Только Слой 3 (human gate) или будущий ревью архитектора-человека.
2. **Качество кода на уровне идиоматики.** Reviewer и stack-файлы ловят грубые нарушения, но субъективные «можно лучше» — только человек.
3. **Гарантия, что менеджер запустил `validate.py`.** Менеджер тоже LLM-агент и может галлюцинировать о том, что валидатор прошёл. Возможный mitigation: пользователь видит вызов в логах и может перепрогнать вручную.
4. **Защита от лжи в EVIDENCE.** Специалист может написать `[VERIFIED:cmd] uv run pytest → all green`, не запустив. Частично страхует `validate.py --rerun-tests`, но это медленно.
5. **Корректная декомпозиция сложных задач.** `shaper` может неправильно разбить работу — это вылезет на этапе coder, когда обнаружится, что задача не атомарна. Возврат к shaper через BLOCKED.
6. **Конкурентность задач.** Один WORKSPACE = одна задача. Параллельные задачи требуют отдельных `TASK_ID`. Встроенного планировщика нет.

### Когда НЕ использовать

- **Задача занимает <30 минут одному разработчику.** Оверхед оркестрации съест всю экономию.
- **Поисково-исследовательская работа** («разберись, как X влияет на Y»). Команда заточена под определённый, формулируемый результат.
- **Хотфикс в продакшне.** Нужна минута, а не час с тремя подтверждениями.

### Известные failure modes

- **Stack-файл становится god-file.** Для сложных стеков (Django с DRF + Celery + Redis) одного файла мало. Решение: разбивать на под-стеки и подключать выборочно.
- **Кросс-стек задачи (full-stack feature).** Менеджер должен правильно сбрифовать `coder_backend` и `coder_frontend` через общий контракт из `ARCH_PLAN.md`. Без чёткого контракта они разбегутся в реализациях.
- **Большие манифесты на локальных моделях.** 4-битные модели <30B могут не удержать в контексте полный манифест + бриф + стек + артефакты. Решение: lite-варианты манифестов под локальные модели (по запросу — могу сделать).
- **Heretic / abliterated fine-tunes.** Деградирует format discipline; в нашем пайплайне это критично. См. `LOCAL-TESTING.md`.

## Рекомендации

### Для первых прогонов

1. **Начни с `EXAMPLE-001` end-to-end на твоей модели.** Используй те же входные данные («эндпоинт `/healthz`»), сравни результат с эталоном в `.agent_space/EXAMPLE-001/`.
2. **Human gates — все включены.** Дефолтная конфигурация в `gates.yaml` — каждый переход требует подтверждения. Не отключай, пока не наберёшь доверия к модели.
3. **Прогоняй `validate.py` руками после каждого handoff.** Не доверяй слепо менеджеру, что он это сделал.
4. **Bench модель на Smoke 0 из `LOCAL-TESTING.md`** перед серьёзной задачей. Это 5 минут, ловит 80% несовместимостей.

### Для повседневной работы

1. **Отключай gates точечно.** После 5–10 успешных прогонов добавляй в `skip_overrides` те переходы, где validate.py надёжно ловит проблемы (обычно это `CODING → REVIEW`, `REVIEW → TESTING`).
2. **Сохраняй gate на ключевом переходе всегда.** `SHAPING → CODING` — здесь зафиксированы scope, архитектура и декомпозиция; самые дорогие ошибки именно тут. Если не уверен в shaper'е — обязательно проверь его выход глазами.
3. **Делай задачи маленькими.** Tech_lead должен производить T-XX по 1–2 файлам. Большие задачи разваливаются на этапе coder.
4. **`.agent_space/` в git — да или нет — но консистентно.** Если задача коллективная — да, для трассируемости. Если соло — можешь не коммитить.

### Для локальных моделей

См. `LOCAL-TESTING.md` — там полная памятка с моделями для M4 Pro 48GB, smoke-тестами и устранением неполадок.

Краткое: **`Qwen2.5-Coder-32B-Instruct-MLX-4bit`** через LM Studio + Kilo Code v7.3+ — рабочий дефолтный стек.

## Подключение к инструменту агентного программирования

### Claude Code (рекомендуемый — Native mode «из коробки»)

В корне уже есть `.claude/agents/` с 10 wrapper-файлами. Открой проект в Claude Code:

```
> Инициализируй задачу TASK-001: <описание>
```

Главная сессия запустит `manager` через `Task`. Менеджер сам вызовет специалистов как изолированных субагентов. Параллельные задачи можно запустить несколькими `Task` в одном ходе.

Опционально: `.claude/settings.json` для permissions:
```json
{
  "permissions": {
    "allow": ["Bash(uv run pytest:*)", "Bash(uv run ruff:*)", "Bash(pnpm test:*)", "Read", "Grep", "Glob"]
  }
}
```

Подробнее: **`integrations/claude-code.md`**.

### Kilo Code v7.3+ (Native mode с file-scoped permissions)

В корне есть `kilo.jsonc` + `.kilo/agents/`. Открой проект в VS Code — Kilo Code подхватит конфиг автоматически.

В чате:
- В agent picker выбери `manager`.
- Запрос: `Инициализируй задачу TASK-001: <описание>`.

Менеджер вызовет специалистов через `Task` (только у `manager` `task: allow`; специалисты вглубь не делегируют).

Per-agent permissions уже настроены: `shaper` пишет только в `.agent_space/`, `coder_backend` — в `apps/`/`src/`/`tests/`/`config/` + `pyproject.toml`, `reviewer` — только `.agent_space/`, `qa` — только в test-файлы, `devops` — только инфра-папки.

Подробнее: **`integrations/kilo-code.md`**.

### Qwen Code / OpenAI Codex CLI (File mode)

В корне есть `AGENTS.md` — универсальная точка входа. Запусти инструмент в этой директории:

```
> Инициализируй задачу TASK-001: <описание>
```

Менеджер создаст `.agent_space/[TASK_ID]/STATE.json` + `handoffs/` и сформирует первый бриф. Завершит шаг сообщением «бриф для shaper готов в `handoffs/01_shaper_brief.md`».

**Для каждого специалиста — новая сессия**:
```
> Возьми активную задачу как shaper.
```

Shaper прочитает свой манифест и бриф, отработает, запишет отчёт. Возвращаешься в менеджера новой сессией:
```
> Прочитай последний handoff от shaper, прогон validate.py и продолжи.
```

Это эмулирует Native-изоляцию через свежие сессии. Подробнее: **`integrations/qwen-code.md`** и **`integrations/codex.md`**.

### Локальная модель через LM Studio + Kilo Code

Самый практичный сетап для оффлайна:

1. LM Studio с `Qwen2.5-Coder-32B-Instruct-MLX-4bit`, сервер на `http://localhost:1234/v1`.
2. Kilo Code v7.3+ → Settings → Provider: OpenAI Compatible → Base URL: `http://localhost:1234/v1`.
3. Smoke 0 из `LOCAL-TESTING.md` для проверки готовности модели.

Подробная памятка с troubleshooting'ом: **`LOCAL-TESTING.md`**.

### Другие OpenAI-совместимые клиенты

Минимальный путь: положи `.agents/`, `.agent_space/`, `AGENTS.md` в корень проекта. Запусти своего агента в этой директории — он прочитает `AGENTS.md` как root-инструкцию, поймёт, что должен действовать как `manager`, и пойдёт по протоколу. Файл-mode будет работать без модификаций; Native mode требует встроенной поддержки subagent-вызовов.

## Расширение системы

### Добавить нового агента

1. Создай `.agents/<agent_id>.md` по образцу существующих манифестов. Единая структура: `SYSTEM_ROLE → ACTIVATION_PROTOCOL → INPUTS → WORKFLOW_STEPS → OUTPUTS → HANDOFF → CONSTRAINTS`.
2. Добавь строку в `AGENT_REGISTRY` в `.agents/manager.md`. При необходимости — в `ROUTING_MATRIX`.
3. Создай wrapper в `.claude/agents/<agent_id>.md` (frontmatter `name`/`description`/`tools` + тело-указатель на манифест).
4. Создай wrapper в `.kilo/agents/<agent_id>.md` (frontmatter `description`/`mode`/`permission` + тело-указатель).
5. Если нужны новые переходы в lifecycle — добавь их в `.agents/_config/gates.yaml`.

### Добавить новый стек

1. Создай `.agents/stacks/<stack_id>.md` по образцу `backend_python_django.md` или `frontend_react.md`. Frontmatter: `stack_id`, `language`, `framework`, `package_manager`, команды (`linter_cmd`, `test_cmd`, ...).
2. В `shaper.md` ничего менять не нужно — он сам обнаружит новый файл при сканировании `.agents/stacks/`.
3. В задачах указывай `tech_stacks: [<stack_id>]` — менеджер пробросит в брифы.

### Кастомные проверки в `validate.py`

Скрипт построен как набор функций-чекеров. Добавление нового чека:

1. Напиши функцию `check_xxx(handoff_text, workspace, report) -> None`.
2. Вызови её в `main()` после существующих проверок.
3. Используй `report.fail(...)` для критичных, `report.warn(...)` для информативных.

## Соглашения и дисциплина

- **Манифесты `.agents/*.md` — единственный источник правды.** `.claude/agents/` и `.kilo/agents/` — тонкие wrappers, ссылающиеся на манифест.
- **Артефакты задачи строго в `.agent_space/[TASK_ID]/`** — никогда вне.
- **`STATE.json.history` — append-only.** Никогда не редактируй прошлые шаги. Если нужен откат — добавь новую запись с указанием на откат.
- **При работе в File mode** после переключения роли (новая сессия) не полагайся на «помню из прошлого». Полагайся на `STATE.json` и брифы в `handoffs/`.
- **EVIDENCE — серьёзно.** Минимум одна `[VERIFIED]` запись обязательна; ASSUMPTIONS — допустимы, но манагер их подсвечивает.
